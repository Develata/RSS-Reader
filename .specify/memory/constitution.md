# RSS 阅读器项目宪章

## 核心原则

### I. Rust 核心，Dioxus 界面
所有生产代码 MUST 使用 Rust 编写。所有面向用户的客户端 MUST 以 Dioxus 作为主要
界面框架，除非存在已记录的平台限制使其不可行。核心业务逻辑、解析、持久化以及导入
导出行为 MUST 置于 UI 组件之外，以便桌面端、Web 端和 Android 端复用相同的应用
逻辑。这样可以保持性能、限制平台分叉，并维持代码库的可维护性。

### II. 本地优先，单用户数据所有权
产品 MUST 作为单用户纯本地客户端运行。SQLite 是订阅源、文章、已读状态、收藏状态
和本地缓存的唯一权威存储。功能设计 MUST 假定抓取完成后数据可在本地使用，并且正
常阅读流程 MUST NOT 依赖任何远端服务。这样可以让 MVP 保持快速、稳健，并且易于
跨平台使用。

### III. 仅配置同步
远端交换 MUST 仅限于订阅源和用户配置。系统在 MVP 阶段 MUST NOT 实现文章、已读
状态、收藏状态或搜索索引同步。WebDAV、OPML 和本地导入导出 MUST 被视为配置传输
机制，而不是通用同步系统。这一硬边界可以防止过早复杂化，同时保留订阅源和偏好设置
的可迁移性。

### IV. 性能是产品特性
变更 MUST 保持顺滑滚动、快速启动、高效拉取订阅，以及在真实个人库规模下仍具备响
应性的列表筛选。订阅刷新路径在源站支持时 MUST 使用 ETag 或 Last-Modified 之类
的条件请求。数据库访问、渲染和状态更新 MUST 避免无意义分配、过度 clone 的数据流
和不必要的异步分层。性能回退必须被明确说明，因为速度是产品本身的一部分，而不是可
选的后期优化项。

### V. 分层边界，简单演进
代码库 MUST 将 UI、应用服务、领域模型和基础设施关注点分离。UI 代码 MUST 依赖
应用层接口，而不是直接嵌入 SQL、HTTP 或解析逻辑。未来能力扩展 MUST 通过稳定边界
和小而明确的抽象实现，而不是依靠推测性的框架、插件系统、微服务或多用户架构。当更
简单的设计已经满足当前需求时，必须选择更简单的设计。所有外部环境依赖，包括文件
系统、网络、浏览器 storage、平台后台任务和远端服务，MUST 通过 adapter、port 或
capability 进入核心逻辑；核心流程不得直接绑定平台细节。

### VI. Headless 命令面，视觉等价交付
前端所有非显示性行为 MUST 以稳定的 Rust 命令语义为长期归宿，而不是永久附着在某一
版按钮、表格、卡片或 DOM 结构上。GUI、CLI、快捷键和未来的命令面板 SHOULD 逐步
复用同一组动作语义。任何以此为目标的前端重构，在默认情况下 MUST 保持现有视觉与交
互体验等价，除非规格文档显式批准用户可见变化。每完成一个模块后，MUST 使用 Chrome
MCP 执行真实页面验证，并证明桌面与小视口下的核心路径、信息层级、按钮触达与 Console
状态均未退化。

### VII. 骨架边界，真相源，失败可验证
任何功能或架构变更 MUST 先说明它落入既有产品骨架的哪个能力轴、影响哪些模块边界，
再进入实现细节。局部功能不得反向塑造主骨架；若确需改变主骨架、核心对象关系、状态
模型、数据模型或配置模型，MUST 先提交变更分析并获得 USER 明确批准。核心对象、核心
状态和核心配置 MUST 拥有唯一真相源；界面缓存、派生状态和测试 fixture 不得与其并列
成为最终依据。所有会沉淀到数据库、磁盘、浏览器 storage、同步链路或导入导出格式中
的结构 MUST 明确版本、迁移、兼容和回退责任。核心流程 MUST 在设计阶段定义失败条件、
失败传播、用户可见结果、重试或退出边界、幂等或去重策略、并发覆盖规则以及验证方式。
无法验证的治理原则不得作为实现完成的依据。

## 产品边界

MVP 范围被刻意收窄。它 MUST 包含添加 RSS 或 Atom 订阅、拉取并解析订阅、展示订阅
列表、展示文章列表、阅读文章正文、标记已读或未读、收藏或取消收藏、按标题搜索、
OPML 导入导出以及本地持久化。它 MUST NOT 包含推荐系统、社交功能、评论、AI 摘要、
复杂标签、多用户授权、插件平台或厚重的设置页。

任何新功能提案 MUST 明确它属于以下哪一类：

- 单用户本地阅读工作流所必需；
- 为保持性能或可维护性所必需；
- 明确延后到 MVP 之后。

如果不属于以上任一类别，则 MUST 被拒绝或延后。

## 交付流程

每个功能规格和实施计划在开始实现前 MUST 证明其符合核心原则。计划文档 MUST 说明
受影响的分层边界、本地数据影响，以及任何配置导入导出影响。只要涉及 schema 变化，
任务列表 MUST 包含持久化或迁移工作；只要涉及订阅刷新、解析或 UI 响应性，任务列表
MUST 包含相应的验证工作。

涉及状态、数据、配置、导入导出、远端交换或平台能力的计划，MUST 明确唯一真相源、
版本/迁移/回退策略、adapter 或 capability 边界、失败路径、观测点和重复执行策略。
若这些事项不受影响，计划 MUST 明确写出“不影响”。

测试遵循基于风险的原则，而不是教条化要求。凡是影响解析、数据库查询、导入导出或状
态流转的改动，MUST 包含自动化验证。UI 改动 MUST 包含桌面/Web 手工验证路径，必
要时还必须覆盖 Android 触控行为。任何有意违反宪章的情况 MUST 记录在计划中的
Complexity Tracking 部分，并说明具体原因以及被放弃的更简单方案。

凡是涉及 headless active interface 重构的任务，计划与任务文档 MUST 明确：

- 本次重构覆盖的模块边界
- 对应的命令层 / 查询层 / 视图壳影响
- 重构前后要执行的 Chrome MCP 基线路径
- 视觉与体验等价的判定标准

没有这些条目的计划，不得进入实现阶段。

每次 agent 完成一次可交付工作后，MUST 在 `docs/handoffs/` 下新增或更新一份固定
格式的交接记录。该记录 MUST 至少包含：工作摘要、受影响模块、受影响平台、关键变更、
验证/验收步骤与结果、当前状态、风险/待跟进项，以及相关 commit、tag 或 worktree 状
态。未补交接记录的工作，不得视为完整交付。

## 治理

本宪章优先于所有与架构、范围和交付相冲突的本地约定。任何修订都需要以书面形式更新
本文件、说明版本号变更原因，并同步更新受影响的模板。

骨架级变更必须先进入分析与审批流程，而不是直接落到实现。分析至少 MUST 包含：变更
原因、当前骨架被证伪或不足之处、收益、风险、对模块/状态/数据/配置模型的影响、迁移
和回退方案、验证方式，以及不变更的长期代价。评审者有责任对 USER 的骨架级判断提出
客观复核意见；复核不削弱 USER 决策权，只用于防止证据不足的方向被固化。

版本策略：

- MAJOR：删除或重定义核心原则，或实质性扩大项目范围。
- MINOR：新增原则、章节或强制工作流要求。
- PATCH：仅做不改变治理意图的澄清性修改。

每个功能计划评审和实现评审都必须执行合规检查。评审者 MUST 确认工作保持 Rust 优
先、本地优先、仅配置同步、性能敏感，并且遵守正确的分层边界。评审者还 MUST 确认本
次工作是否已经补齐 `docs/handoffs/` 交接记录。若本次工作属于 headless active
interface 重构，评审者还 MUST 确认 Chrome MCP 的模块级视觉等价验证已经完成并被
记录。
