# 任务列表：[FEATURE NAME]

**输入**：来自 `/specs/[###-feature-name]/` 的设计文档
**前置条件**：plan.md（必需）、spec.md（用户故事必需）、research.md、data-model.md、contracts/

**测试**：如果改动涉及解析、持久化、导入导出或状态流转，需包含自动化测试。如果改动
涉及 UI 行为，需包含桌面/Web 和 Android 交互的手工验证任务。

**组织方式**：任务按用户故事分组，以便每个故事都能独立实现和测试。

## 格式：`[ID] [P?] [Story] Description`

- **[P]**：可并行执行（不同文件、无依赖）
- **[Story]**：任务所属的用户故事（例如 US1、US2、US3）
- 描述中必须包含精确文件路径

## 路径约定

- **应用层 UI**：`crates/rssr-app/src/`
- **应用服务层**：`crates/rssr-application/src/`
- **领域层**：`crates/rssr-domain/src/`
- **基础设施层**：`crates/rssr-infra/src/`
- **测试**：`tests/` 或 `plan.md` 中定义的各 crate 本地测试

## 依赖与执行顺序

### 阶段依赖

- **初始化（Phase 1）**：无依赖，可立即开始
- **基础能力（Phase 2）**：依赖初始化完成，并阻塞所有用户故事
- **用户故事（Phase 3+）**：都依赖基础能力完成
  - 在有人力时可并行推进
  - 也可按优先级顺序推进（P1 → P2 → P3）
- **打磨阶段（最终阶段）**：依赖所有目标用户故事完成

### 用户故事依赖

- **用户故事 1（P1）**：在基础能力完成后即可开始，不依赖其他故事
- **用户故事 2（P2）**：在基础能力完成后即可开始，可与 US1 集成，但必须保持可独立验证
- **用户故事 3（P3）**：在基础能力完成后即可开始，可与 US1/US2 集成，但必须保持可独立验证

### 每个用户故事内部顺序

- 如果变更涉及持久化、解析或状态逻辑，在关闭故事前必须补齐自动化验证
- 先做领域层/基础设施层改动，再做应用层接线
- 先完成应用层接线，再做 UI 集成
- 先完成核心实现，再做打磨
- 当前故事完成后再进入下一个优先级

### 并行机会

- 所有标记为 [P] 的初始化任务都可并行执行
- 所有标记为 [P] 的基础任务都可在 Phase 2 内并行执行
- Phase 2 完成后，多个用户故事可在团队能力允许时并行开始
- 用户故事中标记为 [P] 的验证任务可并行执行
- 用户故事中标记为 [P] 的模型或支撑任务可并行执行
- 不同用户故事可由不同成员并行推进

## 实施策略

### 先做 MVP（仅用户故事 1）

1. 完成 Phase 1：初始化
2. 完成 Phase 2：基础能力（关键，会阻塞所有故事）
3. 完成 Phase 3：用户故事 1
4. **停止并验证**：独立测试用户故事 1
5. 如果准备就绪，进行演示或交付

### 增量交付

1. 完成初始化 + 基础能力 → 基础就绪
2. 增加用户故事 1 → 独立测试 → 演示/交付（MVP）
3. 增加用户故事 2 → 独立测试 → 演示/交付
4. 增加用户故事 3 → 独立测试 → 演示/交付
5. 每个故事都在不破坏已有能力的前提下增加价值

### 并行团队策略

多人协作时：

1. 团队共同完成初始化和基础能力
2. 基础能力完成后：
   - 开发者 A：用户故事 1
   - 开发者 B：用户故事 2
   - 开发者 C：用户故事 3
3. 各故事独立完成并集成

## 备注

- [P] 任务表示不同文件、无依赖，可并行执行
- [Story] 标签用于将任务映射到具体用户故事，便于追踪
- 每个用户故事都应可独立完成并独立测试
- 在关闭故事前，必须完成必要的自动化和手工验证
- 每个任务或一个合理逻辑组完成后提交
- 在每个检查点停下并独立验证当前故事
- 避免：模糊任务、同文件冲突、破坏独立性的跨故事依赖
