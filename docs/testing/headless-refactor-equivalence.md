# Headless 重构视觉等价验收

## 目的

这份文档定义 headless active interface 重构期间的模块级验收流程。

目标不是验证“功能还能跑”，而是验证：

- 功能仍能跑
- 视觉没有漂移
- 体验没有退化
- 重构没有偷偷改变产品

## 适用范围

只要一次改动满足以下任一条件，就必须执行本流程：

- 提取前端命令层
- 提取查询层 / 视图模型层
- 重排页面组件树
- 把具体按钮行为改为统一分发
- 统一 GUI / CLI 命令语义
- 修改导航壳、阅读页、订阅页、设置页的行为组织方式

## 强制原则

每完成一个模块后，MUST：

1. 用 Chrome MCP 做真实页面验证。
2. 使用与修改前相同的路径和步骤复测。
3. 证明界面视觉与体验没有偏离。

未完成这三项，不得视为模块重构完成。

## 视觉等价的含义

默认情况下，“视觉等价”指：

- 页面标题、主信息和主操作仍出现在相同层级
- 核心组件顺序不发生可感知变化
- 信息密度不意外变稀或变挤
- 用户完成同一任务的点击/滚动/切换成本不增加
- 不新增遮挡、跳动、白屏、错位或闪烁

允许的变化只包括：

- DOM 内部结构调整
- 为了命令层抽象而引入的不可见重构
- 不影响体验的 class / `data-*` 调整

## 模块级执行步骤

### 1. 建立基线

在改模块前，用 Chrome MCP 记录至少以下内容：

- 目标路由
- 桌面视口截图或快照
- 小视口截图或快照
- 关键操作路径
- Console 状态

### 2. 执行同路径复测

改完后，必须按完全相同路径复测：

- 同一路由
- 同一关键操作
- 同一视口
- 同一数据前提

### 3. 验证项目

至少检查：

- 路由进入正常
- 主操作按钮可触发
- 状态提示正常
- 筛选 / 导航 / 保存 / 删除等行为正常
- Console 无新增错误
- 布局无新增错位

### 4. 记录结果

每个模块完成后，MUST 在 handoff 中记录：

- 验证的页面
- 验证的操作
- 视口
- 结果
- 若发现偏离，是否回滚或修正

## 每页最小检查集

### 订阅页

- 添加订阅
- 刷新全部
- 刷新单个订阅
- 删除订阅
- 导入 / 导出配置

### 文章页

- 搜索
- 状态筛选
- 来源筛选
- 分组切换
- 进入阅读页

### 阅读页

- 返回
- 标记已读
- 切换收藏
- 上一篇 / 下一篇导航
- 正文区域显示

### 设置页

- 保存设置
- 应用 CSS
- 主题切换
- WebDAV 上传 / 下载
- GitHub 入口

## 视口要求

每个模块至少检查两个视口：

- 桌面宽度
- 小视口（例如 `390 x 844`）

如果本次模块明确影响移动交互，必须额外检查：

- 返回路径
- 折叠导航
- 点击热区

## Console 门禁

以下情况一律视为未通过：

- panic
- `unreachable`
- 未处理 promise / future 错误
- 新增表单结构告警
- 新增严重样式 / 布局错误
