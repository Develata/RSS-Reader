# 功能设计哲学

## 背景

RSS-Reader 的前端不是一个“什么都能加”的壳，而是一个围绕阅读工作流收束边界的界面。
界面需要同时满足四个目标：

1. 功能边界长期稳定，不因为新想法而不断膨胀。
2. 阅读体验优先，允许为了人类直接阅读而做必要优化。
3. 样式高度可定制，但不牺牲业务稳定性。
4. 在追求体验和性能时，尽量保持多平台兼容。

因此，本项目采用以下总原则：

- **功能有且仅有订阅、阅读、基本设置、基础配置交换。**
- **Rust 控制行为，CSS 控制呈现。**
- **阅读体验优先，但不以牺牲太多多平台兼容性为代价。**
- **GUI 和 CLI 是同一产品能力的不同入口，而不是不同产品。**

---

## 产品边界

### 功能有且仅能有

RSS-Reader 的长期产品范围只围绕四类能力展开。

#### 1. 订阅

订阅能力包括：

- 添加 RSS / Atom 订阅
- 删除订阅
- 手动拉取订阅
- 在应用启动后和后台运行期间按用户偏好自动拉取订阅
- 展示订阅状态与基础信息
  - 名称
  - URL
  - 文章数量
  - 最近刷新状态或失败信息

#### 2. 阅读

阅读能力只服务于“人类直接阅读”。

允许加入的功能，必须直接帮助阅读体验，例如：

- 阅读页排版优化
- 按来源组织和浏览文章
- 按发布时间组织和浏览文章
- 对较旧文章做自动归档，降低当前阅读列表噪音
- 上一篇 / 下一篇导航
- 已读 / 收藏
- 标题搜索与状态筛选
- 正文缓存
- 正文图片本地化
- 键盘阅读流
- 触控设备上的直接点击导航

不应把“阅读”扩展为内容加工平台。

归档在这里被视为阅读组织能力，而不是存储管理能力：

- 归档的目的是降低当前阅读面中的旧内容噪音
- 归档后的文章仍然保留在本地库中，可再次查看和阅读
- 自动归档阈值应可在设置页中调整，默认值为 3 个月

#### 3. 基本设置

设置能力保持克制，但必须高度支持样式自定义。

优先支持：

- 编辑自定义 CSS
- 导入 `.css` 主题文件
- 导出当前 CSS
- 切换官方内置主题
- 保存阅读相关偏好和刷新偏好

不把设置页发展成复杂的可视化设计器。

#### 4. 最基础的同步 / 导入 / 导出

配置交换能力只围绕：

- 订阅链接
- 用户偏好设置

当前允许的交换形式：

- 配置包 JSON
- OPML
- WebDAV 配置上传 / 下载

兼容性保真字段允许存在，但不应反客为主成为核心产品能力。例如 OPML 中常见的分组 /
`folder` 信息可以继续被保留、导入、导出，以降低与外部工具互通时的信息损失；但它只
是配置交换层的保真字段，不应扩展成 GUI 主线中的文件夹系统、管理树或人工组织体系。

明确排除：

- 文章库同步
- 阅读历史同步
- 收藏历史同步
- 跨设备冲突合并

### 禁止增加无关功能

RSS-Reader 明确不增加与上述四类能力无关的功能，尤其是：

- AI 分析
- AI 总结
- 文本改写
- 推荐流
- 评论 / 社交功能
- 插件商店式能力膨胀

如果某个新需求不能直接落回“订阅、阅读、基本设置、基础配置交换”中的一类，它就应被视为越界。

## 体验、性能与兼容性

### 阅读体验优先

- 产品的首要体验目标是尽快让用户开始阅读。
- 一切新增交互都应优先判断：它是否降低了从“打开应用”到“开始阅读”的摩擦。
- 允许为了阅读舒适度增加必要能力，但这些能力必须直接服务阅读，而不是转移产品重心。
- 可以借鉴 VitePress / Vue 生态中“文档即阅读界面”的排版优点，例如更清晰的层级、节奏和
  留白，但不应把阅读器做成文档站壳或 CMS 式信息架构。

### 性能是产品特性

- 列表、筛选、搜索、切页和刷新都属于核心体验的一部分。
- 性能优化不是锦上添花，而是阅读器的基础要求。
- 但性能优化应优先选用不破坏多平台兼容性的方案。

### 多平台兼容性不可轻易牺牲

RSS-Reader 的交付形态应长期保持：

- Desktop GUI
  - Windows
  - macOS
- Web GUI
- Mobile GUI
  - 当前以 Android 作为首个稳定落地点
- CLI
- Docker Compose 部署形态
  - 面向 GitHub 生成镜像的 Web 部署

这意味着：

- 不应为了某一平台的炫技效果，把其它平台逼入高维护成本。
- 新能力如果严重依赖单平台特性，需要先证明不会显著拖坏其它交付形态。

## 外部产品借鉴原则

RSS-Reader 可以向成熟 RSS 产品学习，但借鉴对象应始终限定在“订阅体验”和“阅读体验”两类
核心问题上，而不是把别人的产品边界整体搬进来。

### 可以借鉴什么

#### FeedDemon 值得学习的地方

- 围绕“连续阅读”组织界面，而不是围绕功能堆叠组织界面
- 用更低摩擦的桌面阅读流承接“刷新后马上读”
- 让来源视角和时间视角都服务于快速进入正文，而不是制造额外管理负担

#### FreshRSS 值得学习的地方

- 订阅状态展示足够完整
  - 名称
  - URL
  - 刷新状态
  - 错误反馈
  - 文章规模感知
- 在大量订阅源下仍保持清楚、克制的信息组织
- 导入导出、OPML 与同步入口保持务实，而不是做成复杂的平台化能力

#### Fluent Reader 值得学习的地方

- 阅读界面现代、低干扰，适合长文阅读
- 文章列表与阅读页之间的切换摩擦较低
- 视觉完成度高，但仍然保留“阅读器而不是内容平台”的基本形态

### 明确不借鉴什么

- 不借鉴 AI 摘要、AI 分析、推荐流、内容加工等越界能力
- 不借鉴复杂标签树、规则引擎、社交层或评论层
- 不把 OPML / 配置交换中的 `folder` 保真字段演化成 GUI 主线里的文件夹产品能力
- 不借鉴把阅读器做成信息门户、内容平台或 CMS 的方向
- 不因为某个成熟产品已有某功能，就默认它适合 RSS-Reader

### 借鉴时的判断准则

每个新想法在借鉴外部产品之前，都应先回答：

1. 它是否直接改善订阅管理？
2. 它是否直接改善进入阅读和持续阅读的体验？
3. 它是否仍然保持设置页克制，而不是把设置做成系统控制台？
4. 它是否不会破坏多平台兼容性？

如果其中任一答案不清楚，就不应因为“别的产品有”而进入主线。

## 行为与样式边界

### Rust 控制行为

Rust 负责：

- 命令执行
- 数据读写
- 页面导航
- 状态流转
- 错误处理
- 正文缓存与资源本地化

Rust 不负责：

- 主题视觉风格
- 颜色、阴影、圆角、字体等纯呈现决策

### CSS 控制呈现

CSS 负责：

- 布局
- 视觉层级
- 间距
- 字体
- 颜色
- 显隐和重排

CSS 不负责：

- 新增或删除业务命令
- 决定命令是否执行
- 替代路由逻辑
- 存储业务状态

### 命令是公开能力，不是临时点击逻辑

用户可见的核心操作都应是稳定命令，而不是散落在 UI 事件里的临时逻辑。

设计约束：

- 每个命令都应有稳定语义
- UI 与 CLI 共用同一套应用服务
- 前端按钮不通过 spawn CLI 子进程来完成业务

当前对外命令面、页面接口和语义标记已单独整理在：

- [前端命令与界面接口清单](./frontend-command-reference.md)

## 设置与主题哲学

### 样式自定义是“一等能力”

RSS-Reader 不把主题看成装饰性功能，而把它视为设置能力的一部分。

因此：

- 默认样式必须开箱即用
- 官方主题必须可直接切换
- 用户必须能导入和导出 CSS
- 自定义 CSS 必须在持久化和配置交换里被保留

### 样式接口必须稳定

项目不承诺所有内部 DOM 永远不变，但承诺一组稳定的公开样式接口：

- 页面级命名空间
- 导航语义标记
- 命令语义标记
- 稳定组件 class
- CSS 变量

主题作者和 AI 应优先依赖这些接口，而不是 DOM 层级猜测。

更细的接口和 AI 主题生成约束见：

- [主题作者选择器参考](./theme-author-selector-reference.md)

## 阅读缓存与资源策略

### 正文缓存边界

- feed 提供多少正文，就缓存多少正文
- 若 feed 提供全文 HTML，则缓存全文
- 若 feed 只提供摘要，则缓存摘要
- 不默认抓取站点原网页来补全文

### 正文静态资源本地化

为了提升离线可读性和图片加载体验，允许对 feed 正文中引用的静态图片资源做本地化。

当前约束：

- 这类本地化属于阅读增强，而不是独立产品功能
- 它不应阻塞“新增订阅 / 刷新订阅”主流程
- 刷新主流程成功后，可由后台补偿任务继续处理图片本地化
- 一旦本地化失败，阅读器仍然应保留原正文，而不是让刷新整体失败

## 当前实现状态

截至当前版本，这套设计已部分落地，并保持以下现实边界：

- GUI 与 CLI 共用同一套 Rust 应用服务语义
- 自定义 CSS、主题导入导出、官方主题切换已经落地
- Web、Desktop 和 Android 都共享“正文缓存边界不超出 feed 已提供内容”的策略
- 桌面端 / Android 的正文图片本地化以后台补偿任务执行，不阻塞刷新主流程
- Docker Compose 形态服务于 Web 部署，不改变产品功能边界
