# Headless Active Interface 设计目标

## 目的

这份文档定义 RSS-Reader 前端从“语义化 UI + 稳定选择器”演进到“完全 headless active
interface”时的目标架构、非目标、迁移约束和验收门禁。

它回答的是：

- 什么叫 `headless active interface`
- 为什么现有前端还不算完全 headless
- 重构后哪些东西必须稳定，哪些东西可以自由重排
- 为什么这会支持极端 CSS 视觉重构
- 每完成一个模块后，怎样证明视觉和体验没有漂移

## 核心定义

### active interface

active interface 指一组稳定、可调用、可组合的前端活动接口。它们表达的是用户可触发的
业务动作，而不是某个具体按钮、卡片或 DOM 结构。

例如：

- 添加订阅
- 删除订阅
- 刷新单个订阅 / 刷新全部订阅
- 标记已读 / 切换收藏
- 导入配置 / 导入 OPML / 下载 WebDAV 配置
- 打开阅读页 / 返回上一页 / 跳到上一篇

### headless

headless 指这些活动接口不依附于某一种具体视图结构。

它们可以被以下任意载体触发：

- 页面按钮
- 表格单元格
- 工具栏
- 右键菜单
- 快捷键
- 命令面板
- CLI

因此，UI 只是动作的渲染层，而不是动作的定义层。

### view shell

view shell 指负责路由、布局、主题、动画、焦点管理和信息呈现的界面外壳。

它负责：

- 页面切换
- 组件布局
- CSS class / `data-*` 暴露
- 视觉状态展示

它不负责：

- 直接实现业务动作
- 在点击闭包里临时拼接业务规则
- 把命令语义藏在 DOM 结构里

## 当前状态与目标状态

### 当前状态

当前前端已经具备：

- 稳定的 `data-page` / `data-nav` / `data-action`
- 相对清晰的页面和服务边界
- CSS 可对现有 DOM 做较激进重排

但仍存在：

- 大量业务动作直接写在页面组件闭包里
- 同一类动作在不同页面重复接线
- 动作本身没有成为统一的 Rust 命令模型

因此，当前状态更准确地说是：

- 语义化 UI
- 稳定选择器
- 半公开的动作语义

而不是：

- 完全 headless 的命令面

### 目标状态

重构完成后，前端应满足：

1. 所有非显示性动作都先定义为统一的 Rust 命令接口。
2. 页面组件只负责：
   - 收集输入
   - 分发命令
   - 订阅结果状态
3. 同一动作可被 GUI、CLI、快捷键和未来命令面板复用。
4. CSS 可以自由把动作入口重排为：
   - 表格
   - 单框控制台
   - 仪表盘
   - 极简工作台
5. 任何模块级重构都不得改变现有用户可见视觉与体验，除非规格明确批准。

## 目标架构

## 命令层

所有非显示性行为都应首先建模为稳定命令。

最低要求：

- 每个命令只表达一个清晰语义
- 命令名称与 UI 无关
- 命令不以按钮文本、DOM 结构或页面位置命名

推荐命令族：

- FeedCommand
- EntryCommand
- SettingsCommand
- ConfigExchangeCommand
- NavigationCommand
- UiShellCommand

## 查询层

除动作外，稳定查询接口也应与视图解耦。

例如：

- 订阅列表查询
- 阅读页导航查询
- 当前筛选结果查询
- 设置快照查询

查询接口应保证：

- 返回稳定视图模型
- 不把数据库结构直接暴露给视图壳

## 视图壳

视图壳应只负责：

- 路由
- 布局
- 可访问性
- 焦点与快捷键
- 视觉状态

视图壳可以非常自由地重排：

- 按钮可以排成表格
- 订阅信息可以变成单行
- 多页内容可以在视觉上合并成统一工作台

但不得：

- 绕过命令层直接实现业务
- 用 CSS 推断业务状态

## 为什么这会支持极端 CSS

一旦动作先于 UI 被建模，CSS 的自由度会显著提高。

届时可以安全支持：

- 把“文章 / 订阅 / 设置”做成极窄页签
- 把文章列表、订阅状态、设置入口压缩成单框工作台
- 用表格、终端风或密集 dashboard 重排同一套功能
- 让单个动作在不同布局中复用，而不需要重新发明行为

也就是说：

- CSS 决定“看起来像什么”
- Rust 命令层决定“动作是什么”

## 迁移非目标

本次重构明确不做：

- 改变产品边界
- 改变路由语义
- 改变业务结果
- 顺便新增 AI / 社交 / 推荐 / 规则引擎
- 以重构名义重做视觉风格

如果某次模块重构顺便改变了视觉、交互节奏或信息层级，它必须先进入规格，而不是作为默
认副作用混入实现。

## 模块级验收门禁

每完成一个模块，必须按以下门禁通过后才可继续：

1. 命令层和视图壳边界已经清楚。
2. 自动化检查通过。
3. Chrome MCP 对重构前后的真实页面执行同路径验证。
4. 视觉与体验无偏离。

“视觉与体验无偏离”至少包括：

- 页面层级不漂移
- 主要信息顺序不漂移
- 操作触达成本不升高
- 路由切换、按钮反馈、错误提示、筛选与导航行为不回退
- 桌面与小视口下都不出现新增破坏

## Chrome MCP 基线要求

Chrome MCP 验收不是可选附加项，而是模块收口条件。

每个模块至少需要：

- 记录重构前基线
- 运行重构后同路径验证
- 检查 Console
- 检查关键按钮、路由与状态提示
- 检查桌面和一个小视口

统一流程见：

- [Headless 重构视觉等价验收](../testing/headless-refactor-equivalence.md)

## 推荐迁移顺序

建议按以下顺序推进：

1. 订阅页命令面
2. 阅读页命令面
3. 设置与配置交换命令面
4. 导航壳与查询层
5. CLI 与 GUI 的命令统一

这样做的原因是：

- 订阅页动作最集中
- 阅读页动作最容易复用
- 设置页最容易做出统一命令层
- 最后再统一 UI 壳，风险最低
