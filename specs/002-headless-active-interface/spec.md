# 功能规格：完全 Headless Active Interface 重构

**功能分支**：`002-headless-active-interface`  
**创建日期**：2026-04-07  
**状态**：草稿  
**输入**：用户描述：“将当前前端重构为完全的 headless active interface 设计；不要求兼容旧前端内部实现，但必须严格保持当前界面视觉与体验不偏离，并且每重构完一个模块都要用 Chrome MCP 做实际验证。”  

## 用户场景与测试 *(必填)*

### 用户故事 1 - 保持当前使用体验不变地完成命令层抽象（优先级：P1）

作为当前 RSS-Reader 的实际使用者，我希望订阅、阅读、设置和配置交换的现有界面在重构
过程中保持外观和体验不变，同时底层动作逐步收敛到统一的 headless 命令面，这样我可以继
续正常使用应用，而团队可以获得更强的长期可维护性与 CSS 自由度。

**为什么是这个优先级**：如果重构先破坏当前视觉与体验，这次工程会直接违背项目的使用者
利益，也会让后续任何 headless 改造失去可信基线。

**独立测试方式**：对单个被重构模块，在修改前后执行完全相同的 Chrome MCP 路由、操作
和小视口复测，确认视觉层级、主操作触达与结果反馈保持等价。

**验收场景**：

1. **Given** 某个模块尚未重构，**When** 团队为其引入命令层 / 查询层抽象，**Then** 重构
   后的页面视觉层级、主信息顺序和主要交互路径保持不变。
2. **Given** 同一条核心路径在重构前已记录 Chrome MCP 基线，**When** 模块重构完成，
   **Then** 团队可以按相同路径完成复测，并确认没有新增 Console 错误或交互退化。
3. **Given** 订阅、阅读、设置或配置交换中的任意主动作，**When** 其底层迁移到统一命令
   面，**Then** 用户可见结果与原版本保持一致。

---

### 用户故事 2 - 允许极端 CSS 重排而不碰业务行为（优先级：P2）

作为希望强力自定义界面的高级用户或主题作者，我希望前端动作不再依赖当前卡片、列表或
按钮结构，而是依赖稳定的活动接口，这样我可以把界面重排成表格、单框工作台或其它激进
布局，而不需要改 Rust 业务逻辑。

**为什么是这个优先级**：这是这次重构的核心长期价值，但它建立在用户故事 1 的“视觉不回
退、行为不回归”基础上。

**独立测试方式**：对已完成 headless 化的模块，验证其对外仍暴露稳定动作语义、路由语义
和页面命名空间，并确认 CSS 可以只通过公开接口重排现有布局。

**验收场景**：

1. **Given** 某个模块已经完成命令层抽象，**When** 主题作者只依赖公开选择器和命令语义，
   **Then** 可以在不改业务逻辑的前提下重排其视觉布局。
2. **Given** 同一业务动作在 GUI、快捷键或未来命令面板中复用，**When** 不同载体触发它，
   **Then** 动作语义和结果保持一致。

---

### 用户故事 3 - 统一 GUI 与 CLI 的动作语义（优先级：P3）

作为维护者，我希望 GUI 和 CLI 长期收敛到同一套动作语义和查询语义，这样后续可以让 CLI
 变成很薄的命令行外壳，而不是继续维护分散的交互实现。

**为什么是这个优先级**：这是长期收益最大的结果之一，但应晚于核心 GUI 模块的安全重构。

**独立测试方式**：对已抽象出的命令族和查询接口，验证 GUI 与 CLI 都能通过同一语义层表
 达动作，而不是各自保存一套业务入口。

**验收场景**：

1. **Given** 某类动作已经被定义为统一命令，**When** GUI 和 CLI 分别触发该动作，**Then**
   它们共享同一业务语义而不是各自实现一套分叉逻辑。
2. **Given** 维护者需要新增一个动作入口，**When** 该入口被添加到 GUI 或 CLI，**Then**
   只需接入现有命令面，而不需要复制业务规则。

### 边界情况

- 某个模块在重构前就存在布局脆弱性，容易被误判为重构引入问题。
- 命令层抽象后，如果状态更新粒度变粗，可能导致额外重渲染。
- 某个动作本身是页面局部行为，若被错误地全局化，可能增加复杂度。
- 某个模块在视觉上没有变，但焦点路径、按钮触达成本或移动端交互发生细微退化。
- CLI 与 GUI 在同名动作上的输入/输出模型不一致，导致“统一语义”流于表面。

## 需求 *(必填)*

### 功能需求

- **FR-001**：系统 MUST 将前端所有非显示性动作的长期归宿定义为稳定的 Rust 命令语义，而
  不是永久附着在具体按钮、卡片、列表或 DOM 结构上。
- **FR-002**：系统 MUST 为订阅、阅读、设置、配置交换和导航动作定义清晰的命令族或等价
  抽象边界。
- **FR-003**：系统 MUST 为重构中的页面读取路径定义稳定的查询接口或视图模型边界，而不
  直接把底层存储模型暴露给视图壳。
- **FR-004**：系统 MUST 允许视图壳在不改变业务语义的前提下自由重排布局、信息区块和控
  件容器。
- **FR-005**：系统 MUST 保持当前对外产品边界不变，不得借架构重构新增 AI、社交、标签树、
  规则引擎或其它越界能力。
- **FR-006**：系统 MUST 在每完成一个重构模块后，使用 Chrome MCP 对重构前后的真实页面执
  行同路径验证。
- **FR-007**：系统 MUST 默认把“视觉与体验不偏离”视为模块完工门禁；若某次重构需要改变
  用户可见行为，必须先由规格显式批准。
- **FR-008**：系统 MUST 在模块级 Chrome MCP 验证中覆盖至少一个桌面视口和一个小视口。
- **FR-009**：系统 MUST 在模块级 Chrome MCP 验证中检查 Console，不允许新增 panic、
  `unreachable`、严重表单结构告警或关键布局错误。
- **FR-010**：系统 MUST 保持当前路由语义、主要信息层级和主操作触达成本等价，除非规格显
  式批准用户可见变化。
- **FR-011**：系统 SHOULD 让 GUI、CLI、快捷键和未来命令面板逐步复用同一套命令语义。
- **FR-012**：系统 MUST 在设计文档、计划文档和任务文档中明确记录每个模块的命令层 / 查
  询层 / 视图壳边界。
- **FR-013**：系统 MUST 在 handoff 中记录每个模块的 Chrome MCP 验证路径与结果。

### 关键实体 *(若功能涉及数据则填写)*

- **前端命令**：表达单一业务动作的稳定语义接口，不依赖具体视图布局。
- **前端查询接口**：表达页面读取需求的稳定语义接口，向视图壳提供一致的读取模型。
- **视图壳**：负责路由、布局、主题、焦点和视觉状态的界面层，不承担业务动作本体。
- **模块级验收基线**：记录某个模块在重构前后的 Chrome MCP 路由、操作路径、视口和预期结果。

## 产品与边界契合度 *(必填)*

### 本地优先影响

- 本次重构不改变本地优先原则，不改变 SQLite / 浏览器本地状态作为权威来源的现状。
- 重构只改变前端动作与查询的组织方式，不改变本地阅读数据的权威边界。

### 配置交换影响

- 本次重构不扩大配置交换范围。
- 配置导入导出、OPML 与 WebDAV 继续只处理订阅源与用户设置。

### 架构演进影响

- 本次工作直接影响命令层、查询层、视图壳和 GUI / CLI 语义统一。
- 本次工作不会把页面结构变更误写成产品功能变更。

### 非目标与排除项

- 本次重构不改变产品功能边界。
- 本次重构不引入新的用户可见产品能力。
- 本次重构不默认重做视觉风格。
- 本次重构不把“视觉自由”误解成“允许体验漂移”。
- 本次重构不以“统一命令面”为名删除 CLI 入口程序。

## 体验约束 *(必填)*

- 重构必须保持当前页面视觉层级、主信息顺序和主操作位置的整体感知等价。
- 重构必须保持从进入页面到完成关键任务的交互成本不增加。
- 重构必须保持桌面端和小视口下的主路径可用。
- 若某模块当前具备明确快捷键、返回路径、目录层级或筛选交互，这些体验必须保持等价。
- 每个模块都必须在重构前后通过 Chrome MCP 基线对照验证。

## 成功标准 *(必填)*

### 可衡量结果

- **SC-001**：对每个完成重构的模块，都存在一条已记录的 Chrome MCP 前后对照验证记录，且结
  果为通过。
- **SC-002**：所有已完成重构模块在桌面与小视口下均无新增视觉漂移、主路径退化或 Console 严
  重错误。
- **SC-003**：订阅、阅读、设置和配置交换四大动作域都被映射到清晰的命令族或等价抽象边界。
- **SC-004**：至少一个 GUI 动作域与 CLI 动作域完成语义统一，不再需要两套独立业务规则。
