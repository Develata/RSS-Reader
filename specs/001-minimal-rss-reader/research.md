# 研究记录：极简个人 RSS 阅读器 MVP

## 决策 1：采用单一 Rust workspace，而不是“客户端 + 本地服务”双进程结构

**Decision**：采用单一 Rust workspace，并在代码层做 UI / 应用服务 / 领域 / 基础设施分层。

**Rationale**：当前产品是单用户、本地优先的 RSS 阅读器。双进程结构会增加启动成本、
部署复杂度、调试成本和状态同步成本，而不会为 MVP 带来等价价值。四层分离已经足够
支持长期演进，并能在不引入额外进程边界的前提下保持架构清晰。

**Alternatives considered**：
- 客户端 + 本地 API 服务：边界更硬，但过早复杂化。
- 单 crate 全堆叠：启动快，但后续维护与演进成本过高。

## 决策 2：使用 SQLite 作为唯一事实来源

**Decision**：订阅源、文章、已读/收藏状态和偏好设置全部以本地 SQLite 为权威数据源。

**Rationale**：产品明确要求本地优先、单用户、可离线阅读已抓取内容。SQLite 足够承载
个人规模数据，具备成熟索引能力、迁移能力和良好跨平台支持，也便于保持启动快和部署
简单。

**Alternatives considered**：
- 文件型 KV 存储：对复杂查询、筛选和搜索支持弱。
- 嵌入式文档数据库：额外复杂度和收益不匹配。

## 决策 3：同步收缩为“配置交换”，不设计文章同步

**Decision**：仅交换订阅源和用户偏好设置；不交换文章正文、已读状态、收藏状态和搜索索引。

**Rationale**：这是用户已明确固定的边界。这样可以大幅降低冲突处理、版本合并、设备
游标和远端一致性问题，保留 WebDAV/导入导出/OPML 的迁移价值，同时避免系统演变成
完整同步产品。

**Alternatives considered**：
- 完整文章与状态同步：明显超出范围。
- 仅 OPML：无法保存主题和偏好设置，不满足迁移体验要求。

## 决策 4：订阅刷新优先使用条件请求

**Decision**：对每个订阅源保存 `ETag` 与 `Last-Modified`，刷新时优先发送条件请求。

**Rationale**：这能显著降低无变化订阅源的网络与解析开销，直接服务于“快”和“顺滑”
两个目标，并减少对远端源站的无效压力。

**Alternatives considered**：
- 每次全量拉取：实现简单，但浪费明显。
- 复杂后台调度器：首版过重，后续再按需要扩展。

## 决策 5：文章去重采用“源内稳定键 + 内容更新”策略

**Decision**：以订阅源内稳定外部标识为主键依据，辅以链接或归一化标题+发布时间作为
兜底，命中既有文章时执行更新而不是新建。

**Rationale**：RSS/Atom 在真实世界中的 ID 质量并不完全可靠。该策略可以在保证简单
的同时处理大多数重复拉取和源端内容更新场景。

**Alternatives considered**：
- 仅按标题去重：误判率高。
- 全局 UUID 强制重建：无法自然映射源端条目。

## 决策 6：搜索范围限定为标题搜索

**Decision**：MVP 仅支持标题搜索，不做正文全文检索。

**Rationale**：标题搜索已经覆盖用户明确需求，且对本地性能和实现复杂度更友好。全文
搜索会额外引入索引维护、分词或 SQLite FTS 设计，而当前并非必要。

**Alternatives considered**：
- SQLite FTS 全文搜索：未来可加，但首版不需要。
- 无搜索：不满足功能范围。

## 决策 7：导入导出契约使用 JSON 配置包 + OPML

**Decision**：提供两类契约：OPML 只管订阅源导入导出；JSON 配置包负责订阅源和用
户偏好设置的完整迁移与 WebDAV 存储。

**Rationale**：OPML 适合与外部阅读器互通，但无法完整表达偏好设置；JSON 配置包更
适合项目内部稳定演进和远端配置交换。

**Alternatives considered**：
- 仅 JSON：互操作性不足。
- 仅 OPML：表达能力不足。
