# 功能规格：极简个人 RSS 阅读器 MVP

**功能分支**：`001-minimal-rss-reader`  
**创建日期**：2026-03-24  
**状态**：草稿  
**输入**：用户描述：“构建一个极简的个人 RSS 阅读器，使用本地 SQLite 存储，并通过 WebDAV、导入导出和 OPML 进行配置同步”

## 用户场景与测试 *(必填)*

### 用户故事 1 - 高效阅读我的订阅（优先级：P1）

作为个人用户，我希望添加订阅并以快速、低摩擦的方式阅读新文章，这样我可以持续跟踪
信息源，而不会被噪音和不必要的操作打断。

**为什么是这个优先级**：如果用户不能顺畅地订阅、刷新、浏览并从本地阅读文章，这个
产品就没有实际价值。

**独立测试方式**：添加一个或多个 feed URL，刷新本地库，打开文章列表，并在除了抓
取 feed 本身之外不依赖任何远端服务的情况下阅读完整文章内容。

**验收场景**：

1. **Given** 用户当前没有任何订阅，**When** 用户添加一个有效的 feed URL，**Then**
   系统保存该订阅，并在订阅列表中展示它。
2. **Given** 某个已保存订阅包含已发布文章，**When** 用户刷新订阅，**Then** 新文
   章出现在文章列表中，且不会重复写入本地已有文章。
3. **Given** 文章列表中同时包含未读和已读文章，**When** 用户打开一篇文章，**Then**
   阅读视图展示文章标题、来源、发布时间和可阅读的正文内容。
4. **Given** 用户已经配置自动刷新偏好，**When** 应用启动或在后台运行达到刷新间隔，
   **Then** 系统会自动尝试拉取订阅，并向用户展示最新刷新状态。
5. **Given** 用户需要围绕单个来源连续阅读，**When** 用户切换到按来源组织的文章视图，
   **Then** 系统会按订阅源对文章进行分组或分类展示。
6. **Given** 用户需要围绕发布时间整理阅读节奏，**When** 用户切换到按时间组织的文章视图，
   **Then** 系统会按发布日期对文章进行分组或分类展示。
7. **Given** 用户未修改归档阈值，**When** 本地文章发布时间超过 3 个月，**Then**
   系统会自动将其标记为归档并从默认当前阅读面中弱化，但文章仍保留在本地库中可再次阅读。

---

### 用户故事 2 - 管理阅读进度（优先级：P2）

作为个人用户，我希望能将文章标记为已读或未读，并进行收藏或取消收藏，这样我可以跟
踪重要内容，而不必引入复杂的组织系统。

**为什么是这个优先级**：阅读状态是个人 RSS 工作流可用性的关键部分，但它依赖于文
章已经存在于本地。

**独立测试方式**：基于已有本地文章库，对多篇文章切换已读和收藏状态，并验证这些状
态在应用重启后仍然保留。

**验收场景**：

1. **Given** 列表中有一篇未读文章，**When** 用户将其标记为已读，**Then** 列表立
   即反映新状态，并且在重新打开应用后仍保持该状态。
2. **Given** 一篇文章出现在列表或阅读页中，**When** 用户收藏它，**Then** 该文章
   会出现在收藏结果中，直到用户取消收藏。
3. **Given** 用户希望回看未读或已收藏内容，**When** 用户应用对应筛选条件，**Then**
   列表只展示符合条件的文章。

---

### 用户故事 3 - 携带我的订阅源和偏好设置（优先级：P3）

作为个人用户，我希望导入、导出，并在可选情况下远端存放我的订阅源和偏好设置，这样
我可以在不同安装环境之间迁移，而不必同步完整文章库。

**为什么是这个优先级**：配置可迁移性很重要，但它仍然排在核心本地阅读工作流之后。

**独立测试方式**：从一个安装环境导出订阅和设置，在另一个安装环境中导入，并确认第
二个环境能重建订阅列表和用户偏好，而不会导入文章历史。

**验收场景**：

1. **Given** 用户已经配置好本地阅读库，**When** 用户导出配置，**Then** 导出结果
   包含恢复阅读环境所需的订阅源和个人偏好设置。
2. **Given** 用户拥有一个 OPML 文件，**When** 用户导入它，**Then** 系统会创建所
   有当前尚不存在的新订阅。
3. **Given** 用户连接了一个远端配置位置，**When** 用户上传或下载配置，**Then**
   交换的数据只包含订阅源和个人偏好设置。
4. **Given** 用户已经在设置页中选择预置主题或导入自定义 CSS，**When** 用户保存设
   置并导出配置，**Then** 这些界面偏好会随用户设置一起迁移到新的安装环境。
5. **Given** 用户在设置页中需要访问项目源码与 issue 入口，**When** 用户点击
   GitHub 仓库图标，**Then** 系统会打开该项目的 GitHub 仓库页面，而不打断当前
   本地阅读数据。

### 边界情况

- 添加的 URL 并未暴露有效的 RSS 或 Atom feed。
- 某次订阅刷新没有变化、只返回部分数据，或出现临时抓取失败。
- 某个信息源重新发布了一篇已存在但内容被编辑过的文章。
- 多个订阅源发布了标题相同的文章。
- 导入的 OPML 或配置文件中包含重复的 feed URL。
- 远端配置文件不可用、过时或格式损坏。
- 某篇文章正文不完整、包含不支持的标记，或缺少发布时间元数据。
- 用户搜索时没有任何匹配结果。
- 用户切换按来源或按时间浏览时，本地库中没有任何符合条件的当前文章。
- 某篇文章缺少发布时间，导致其无法按正常时间桶归类。
- 自动归档阈值被用户调小后，大量旧文章在一次重算中进入归档状态。

## 需求 *(必填)*

### 功能需求

- **FR-001**：系统 MUST 允许用户通过输入 feed URL 添加订阅源。
- **FR-002**：系统 MUST 为每个已保存订阅拉取并存储订阅元数据和文章条目。
- **FR-003**：系统 MUST 允许用户手动刷新全部订阅源，以及刷新单个订阅源。
- **FR-003a**：系统 MUST 支持在应用启动后和后台运行期间按用户偏好自动拉取订阅，
  同时保留用户可随时触发的手动刷新入口。
- **FR-004**：系统 MUST 在刷新时避免为已存在的本地文章创建重复记录。
- **FR-005**：系统 MUST 提供订阅列表，使用户可以查看全部文章，或将文章列表缩小到
  单个订阅源。
- **FR-005a**：系统 MUST 在订阅视图中展示订阅状态和基础信息，包括订阅名称、URL、
  本地文章数量以及最近刷新状态或失败反馈。
- **FR-005b**：若设置页对用户暴露订阅相关偏好（例如刷新间隔），系统 MUST 让这些偏
  好对运行时行为产生可观察效果，而不是仅作持久化展示。
- **FR-006**：系统 MUST 提供按高效阅读排序的文章列表，并支持打开任意已存储文章
  的完整阅读视图。
- **FR-006a**：系统 MUST 支持按来源对文章进行分类或分组浏览，以便用户围绕单个信息源
  连续阅读。
- **FR-006b**：系统 MUST 支持按时间对文章进行分类或分组浏览，以便用户围绕发布时间组
  织阅读。
- **FR-006c**：系统 MUST 支持自动归档超过阈值的旧文章；该阈值 MUST 可在设置页中配置，
  默认值为 3 个月。
- **FR-006d**：系统 MUST 将归档视为阅读组织状态而不是删除行为；归档文章 MUST 继续保
  留在本地数据库中，并可再次查看、搜索和阅读。
- **FR-007**：系统 MUST 持久化存储本地重新打开并阅读文章所需的正文内容。
- **FR-007a**：系统 MUST 将正文缓存边界限定为 feed 已提供的正文内容；若 feed 提供全
  文 HTML，则缓存全文；若只提供摘要，则缓存摘要，而 MUST NOT 默认抓取站点原网页来补
  全正文。
- **FR-007b**：系统 SHOULD 尽可能将正文中引用的静态图片资源本地化，并在渲染时优先
  使用本地引用，以改善离线可读性和图片加载体验。
- **FR-008**：系统 MUST 允许用户将文章标记为已读或未读。
- **FR-009**：系统 MUST 允许用户对文章进行收藏或取消收藏。
- **FR-010**：系统 MUST 允许按未读状态和收藏状态筛选文章。
- **FR-011**：系统 MUST 允许按文章标题搜索。
- **FR-012**：系统 MUST 在应用重启后保留订阅、文章、已读状态、收藏状态和用户偏
  好设置。
- **FR-013**：系统 MUST 支持订阅源的 OPML 导入和 OPML 导出。
- **FR-014**：系统 MUST 支持订阅源和个人偏好设置的配置导出与导入。
- **FR-015**：系统 MUST 支持连接远端配置位置，并上传或下载订阅源和个人偏好设置。
- **FR-016**：系统 MUST NOT 在配置交换过程中上传、下载或合并文章正文、阅读历史、
  收藏历史或搜索索引。
- **FR-017**：系统 MUST 提供浅色、深色和跟随系统的主题行为。
- **FR-018**：系统 MUST 在桌面类平台上支持键盘友好的阅读流程。
- **FR-019**：系统 MUST 在触控设备上保持主阅读流程可用，并支持直接点击导航和操
  作。
- **FR-020**：系统 MUST 在刷新、导入或远端配置操作失败时向用户提供清晰反馈。
- **FR-021**：系统 MUST 允许用户在设置页中编辑自定义 CSS、导入和导出当前 CSS，
  并通过预置主题按钮、主题下拉和主题卡片快速切换主题。
- **FR-022**：系统 MUST 在设置页中显示当前自定义样式来源，并将 `custom_css`
  作为用户偏好设置的一部分进行持久化与配置交换。
- **FR-022a**：系统若在设置页中暴露列表密度、启动视图或阅读字号等阅读偏好，则这些设
  置 MUST 对界面或启动路径产生真实影响；系统 MUST NOT 长期保留无实际效果的占位设置项。
- **FR-022b**：系统若在设置页中暴露自动归档阈值，则该阈值 MUST 对文章归档行为产生真实
  影响，并在用户修改后可重新计算当前文章的归档状态。
- **FR-023**：系统 MUST 采用紧凑导航头而不是占据大量版面的品牌展示块，并将产品识
  别控制在不干扰阅读与管理任务的位置。
- **FR-024**：系统 MUST 在设置页中提供一个轻量的 GitHub 仓库入口，使用户可以直
  接跳转到项目主页。
- **FR-025**：系统 MUST 将产品功能边界收敛在订阅、阅读、基本设置和基础配置交换之
  内；任何新增能力若不能直接服务这四类目标，则 MUST NOT 进入当前产品范围。
- **FR-026**：系统 MUST NOT 引入文本加工型能力，例如 AI 分析、AI 总结、文本改写
  或其它与直接阅读无关的内容处理流程。

### 关键实体 *(若功能涉及数据则填写)*

- **订阅源**：由用户管理的信息源，包含规范 URL、展示标题、可选分组元数据和刷新元
  数据。
- **文章**：由订阅源发布并存储在本地的条目，包含标题、来源引用、发布时间、正文和
  原文链接。
- **文章状态**：附着在文章上的用户状态，包括已读状态和收藏状态。
- **文章组织状态**：附着在文章上的阅读组织信息，包括归档状态，以及用于按来源和时间浏
  览时所需的分类维度。
- **用户偏好设置**：阅读和界面偏好，例如主题、密度、启动行为、刷新偏好、自动归档阈值和可迁移的
  自定义 CSS。
- **配置包**：一种可迁移导出内容，包含订阅源和用户偏好设置，但不包含文章正文和个
  人阅读历史。

## 产品与边界契合度 *(必填)*

### 本地优先影响

- 该功能将本地阅读库视为用户的主阅读环境。
- 阅读进度、收藏状态、文章缓存和搜索行为都依赖于本地存储数据跨会话可用。
- 远端连接仅用于刷新订阅或交换配置，不用于正常浏览已经抓取到本地的内容。

### 配置交换影响

- 该功能包含订阅源的 OPML 导入导出。
- 该功能包含包含订阅源和用户偏好设置的可迁移配置包导入导出。
- 该功能包含将配置包上传到远端位置或从远端位置下载。
- 该功能明确排除完整文章库同步。

### 功能范围影响

- 产品能力只围绕订阅、阅读、基本设置和基础配置交换展开。
- 为改善直接阅读体验而加入的排版、导航、缓存和图片本地化增强，仍然归属于“阅读”范畴，
  而不是扩展新的产品线。
- 可借鉴 FeedDemon、FreshRSS、Fluent Reader 在阅读流、订阅状态展示和界面完成度上的长
  处，但所有借鉴都必须服从本产品既定边界，而不是演化成更大的信息平台。
- 按来源组织、按时间组织和自动归档属于阅读组织能力，应服务于降低噪音、提升进入阅读的速
  度，而不是引入内容管理系统式复杂度。
- GUI 和 CLI 共享同一产品边界，只是交付入口不同。
- Docker Compose / GitHub 镜像属于 Web 部署形态，不构成新的独立功能范围。
- 浏览器或服务器登录门禁若存在，应被视为部署或本地数据保护包装层，而不是阅读器的核心
  业务能力本身。

### 非目标与排除项

- 本功能不增加推荐、评论、社交分享、AI 摘要、复杂标签、多用户账户或插件生态。
- 本功能不增加 AI 分析、AI 总结、文本改写或其他文本加工型能力。
- 本功能不把浏览器本地登录门禁或服务器登录门禁作为核心产品卖点；它们仅在需要保护 Web
  入口时作为部署包装层存在。
- 本功能不增加远端文章存储、跨设备已读状态同步、跨设备收藏状态同步或文章库冲突解
  决。
- 本功能不把“抓取站点原网页并做全文抽取”作为默认正文缓存策略；正文缓存边界保持在
  feed 已提供的内容之内，静态资源本地化另行按阅读体验优化处理。
- 本功能不增加超出阅读舒适度和配置交换所需范围之外的厚重设置页。
- 本功能不将“来源 / 时间分类”和“自动归档”扩展为复杂标签树、文件夹系统或可编程规则引
  擎；它们只服务于更清晰的阅读组织。
- OPML 或配置包中的 `folder` 信息若被保留，应仅用于导入导出保真与互操作，不应被视为
  当前产品主线中的文件夹能力。
- 本功能不增加实时主题预览模式、所见即所得主题编辑器或独立主题商店式浏览流程。
- 本功能不把 Android 正式签名发布、AAB 产物和商店分发纳入当前 MVP 交付门禁；这些内容
  作为后续发布能力，单独记录在 `docs/roadmaps/android-release-roadmap.md`。
- 当前仓库已补充 Android Debug APK 构建路径与调试产物能力，用于安装验证和移动端后续
  开发，但它不替代正式 Android Release 交付标准。
- 多平台 GUI / CLI 是交付形态目标，但不意味着为每个平台引入不同功能集合；Windows、
  macOS、Web、移动端和 Docker Compose Web 部署应共享同一产品边界。

## 体验约束 *(必填)*

- 核心阅读流程必须尽量减少从启动应用到开始阅读之间的步骤数。
- 即便在较大的个人文章库下，文章浏览也必须保持可读和响应迅速。
- 阅读视图必须优先保证清晰排版和舒适的长文阅读体验。
- 文章组织方式必须优先围绕“来源”和“时间”这两种最直接的阅读维度，而不是引入高维护成本
  的人工分类体系。
- 桌面类平台必须支持以键盘为中心的主阅读操作导航。
- 触控平台必须支持自然的点击目标和直接明了的返回导航。
- 顶部导航必须优先服务于信息访问，不应保留占据明显纵向空间的装饰性品牌头图。
- 界面必须保持克制，支持浅色和深色外观，并避免与内容竞争注意力的装饰性设计。
- 可借鉴 VitePress / Vue 生态中清晰、克制、重排版的阅读界面风格，但不得照搬文档站式站
  点壳层、章节导航或 CMS 化信息结构。

## 假设

- 产品面向的是每个安装环境中由单个用户拥有的一套个人阅读库。
- 配置交换采用单一最新版本覆盖模型，而不是多版本合并模型。
- 对 MVP 而言，标题搜索已经足够；全文搜索不在初始范围内。
- 用户可以接受已读状态和收藏状态保持为各设备本地数据。

## 成功标准 *(必填)*

### 可衡量结果

- **SC-001**：新用户无需外部指导，即可在 3 分钟内完成至少一个有效 feed 的添加，并
  进入可阅读的文章视图。
- **SC-002**：在包含 10,000 篇本地文章的阅读库中，文章列表浏览、筛选切换和标题搜
  索都保持足够响应，使用户在交互时不会感受到明显停顿。
- **SC-003**：在多次刷新未变化的订阅源时，系统不会重复保存已存在文章，且不会给用
  户制造明显噪音。
- **SC-004**：用户可以从一个安装环境导出订阅源和偏好设置，并在另一个安装环境中通
  过受支持的导入路径，以至少 95% 的首次成功率恢复它们。
